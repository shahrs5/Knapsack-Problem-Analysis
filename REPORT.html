Abstract

The 0/10/1 Knapsack Problem is a classic NP-hard combinatorial optimization problem with applications in resource allocation, portfolio selection, and cryptography. The problem is also conceptually linked to practical sales operations, such as the Traveling Salesperson Problem (TSP); before a salesperson departs on their tour, they must solve a knapsack problem to figure out what items or goods to carry, because a salesperson without their goods is not a good salesperson. This report presents a comparative empirical analysis of three core algorithms: Brute Force (Exhaustive Recursion), Dynamic Programming (Bottom-up Tabulation), and a Greedy Approximation Heuristic (Value-to-Weight Ratio Sort). We evaluate these algorithms across synthetic datasets, standard benchmark datasets from the kplib library, and large-scale hard benchmarks. Our findings confirm that Brute Force exhibits exponential time growth O(2n)O(2^n), becoming intractable for n>20n > 20. Dynamic Programming guarantees optimal solutions in pseudo-polynomial time O(nW)O(nW), but suffers from memory constraints when capacity W≥108W \ge 10^8. The Greedy heuristic executes in O(nlog⁡n)O(n \log n) time, achieving an average approximation ratio above 99%99\% on most instances, though its quality degrades to 95.68%95.68\% on strongly correlated datasets where profit is tightly coupled with weight. We analyze these computational trade-offs to guide algorithmic selection.

1. Problem Statement

Given nn items, where each item ii (1≤i≤n1 \le i \le n) has a profit pi>0p_i > 0 and weight wi>0w_i > 0, and a knapsack of capacity W>0W > 0, select a subset of items to maximize the total profit without exceeding the capacity WW. Using binary variables xi∈{0,1}x_i \in \{0, 1\}, where xi=1x_i = 1 if item ii is selected and 00 otherwise, the problem is formulated as: Maximize: ∑i=1npixisubject to ∑i=1nwixi≤W\text{Maximize: } \sum_{i=1}^{n} p_i x_i \quad \text{subject to } \sum_{i=1}^{n} w_i x_i \le W

Computational Complexity

  • Decision Version: Asking if a profit of at least PP can be achieved is NP-complete (proved by reduction from Subset Sum).
  • Optimization Version: Maximizing profit is NP-hard. The search space consists of 2n2^n subsets. However, if WW is bounded, it can be solved in pseudo-polynomial time O(nW)O(nW) using Dynamic Programming.

2. Description of Algorithms Used

2.1 Brute Force (Exhaustive Recursive Search)

  • Technique: Explores the full binary decision tree via recursion.
  • Recurrence Relation: K(i,c)={0if i=0 or c=0K(i−1,c)if wi>cmax⁡(K(i−1,c),pi+K(i−1,c−wi))otherwiseK(i, c) = \begin{cases} 0 & \text{if } i = 0 \text{ or } c = 0 \\ K(i-1, c) & \text{if } w_i > c \\ \max(K(i-1, c), p_i + K(i-1, c - w_i)) & \text{otherwise} \end{cases}
  • Complexity: Time: O(2n)O(2^n), Space: O(n)O(n) due to call stack depth.
def knapsack_brute_force(weights, values, capacity, n=None):
    if n is None: n = len(weights)
    if n == 0 or capacity == 0: return 0
    if weights[n - 1] > capacity: return knapsack_brute_force(weights, values, capacity, n - 1)
    return max(knapsack_brute_force(weights, values, capacity, n - 1),
               values[n - 1] + knapsack_brute_force(weights, values, capacity - weights[n - 1], n - 1))

2.2 Dynamic Programming (Bottom-Up Tabulation)

  • Technique: Tabulation with space optimization. Uses a 1D array dp of size W+1W+1. The capacity loop runs backwards from WW down to wiw_i to ensure each item is selected at most once.
  • Complexity: Time: O(nW)O(nW), Space: O(W)O(W).
def knapsack_dp(weights, values, capacity):
    dp = [0] * (capacity + 1)
    for i in range(len(weights)):
        for w in range(capacity, weights[i] - 1, -1):
            dp[w] = max(dp[w], values[i] + dp[w - weights[i]])
    return dp[capacity]

2.3 Greedy Approximation Heuristic

  • Technique: Sorts items descending by profit-to-weight ratio ri=pi/wir_i = p_i / w_i. The scan adds items until the remaining capacity cannot fit the next item.
  • Complexity: Time: O(nlog⁡n)O(n \log n), Space: O(n)O(n).
def knapsack_greedy(weights, values, capacity):
    order = sorted(range(len(weights)), key=lambda i: values[i] / weights[i], reverse=True)
    total, remaining = 0, capacity
    for i in order:
        if weights[i] <= remaining:
            total += values[i]
            remaining -= weights[i]
    return total

We evaluate the algorithms across four test beds:

  1. Synthetic Datasets: Weights wi∈[1,50]w_i \in [1, 50], values pi∈[1,100]p_i \in [1, 100], and capacity W=⌊0.4∑wi⌋W = \lfloor 0.4 \sum w_i \rfloor.

    • 00Uncorrelated: pip_i and wiw_i are independent.

  4. Synthetic Edge Cases Suite: Verification suite containing boundary conditions (single item, W=0W=0, W≥∑wiW \ge \sum w_i, capacity less than minimum weight, and a classic greedy failure counterexample).

7. Discussion and Analysis

  • Brute Force: Serves as a correctness baseline; limited by its O(2n)O(2^n) exponential complexity.
  • Dynamic Programming: Best for absolute optimality when capacity WW is small to moderate. Suffers from high computational and memory O(W)O(W) limits when W≥108W \ge 10^8.
  • Greedy Heuristic: Runs in O(nlog⁡n)O(n \log n) time with O(n)O(n) space. Highly scalable, making it the practical choice for real-time or massive instances, though susceptible to correlation-induced sub-optimality.

8. Conclusion

We evaluated three Knapsack algorithms across synthetic, standard (kplib), and large-scale hard benchmarks. Our analysis shows that Brute Force is limited to n≤20n \le 20, Dynamic Programming is bound by memory when W≥108W \ge 10^8, and Greedy heuristics scale near-instantly but yield sub-optimal results (95%95\%-99%99\%) depending on dataset correlation. Algorithm selection must balance constraints between optimality requirements and memory budgets.
